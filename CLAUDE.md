# Brookfield Comfort — Internal Platform (CLAUDE.md)

Guidance for Claude working in this repo. This file is the summary of record; `docs/API-RULES.md` holds the full API/client conventions. (The original build spec, `FRONTEND_SPEC.md`, has been retired — its essential domain knowledge and landmines are captured here.)

## What this is

A **modular internal platform** for Brookfield Comfort (UK footwear e-commerce — mostly Birkenstock, sold on Shopify & Amazon). Login → dashboard of modules. **v1 ships ONE module: "Shopify Pricing".** Architect for more modules (Amazon pricing, inventory, orders, analytics — shown as "coming soon" tiles) but **only build Shopify Pricing** unless asked. The shell (auth, dashboard, nav, API envelope, DB pool) is deliberately module-agnostic.

## Structure — two apps in one repo

```
bcweb-server/   Express API — owns the Postgres connection and ALL SQL. Runs on a VPS (PM2).
bcweb-web/      Next.js 15 (App Router, src/app, TS, Tailwind 3) front end. Deploys to Vercel.
docs/                API-RULES.md (conventions, authoritative), deploy.txt
```

The web app **never** connects to Postgres — only the server does. Web → HTTP (axios) → server → PostgreSQL (`brookfield_prod`, the same DB the owner's Python scripts use).

## Run locally (Windows)

```
# API on :3020 — reads bcweb-server/.env (DB_* + JWT_SECRET)
cd bcweb-server && npm run dev
# Web on :3000 — reads bcweb-web/.env (NEXT_PUBLIC_API_URL)
cd bcweb-web && npm run dev
```
Start the API first. Seed/reset a user: `node scripts/seed-user.js <username> <password> Andreas`.
Deployment: `docs/deploy.txt` (server → VPS/PM2 rsync; web → Vercel, public URL behind the app login).

## The Shopify Pricing domain (the important "why")

**Business context:** ~95% of Shopify sales are Birkenstock, which **cannot be re-ordered on demand** (ordered ~6 months ahead; stock in hand is all there is). So there is no "sold out → restock" lever — **the job is to squeeze maximum margin from stock already held.** A fast-selling style with thin stock is a **price-up / harvest** candidate (it'll sell through anyway; a low price just donates margin), NOT a restock flag.

**Process (always starts from a segment).** The segment page has a prominent **WINNERS | LOSERS** switch — two opposite jobs:
1. **Triage / WINNERS (shortlist):** top ~10 in-stock styles by units sold in the last 30 days on Shopify. Drop 0-stock styles (nothing to price; can't restock) and top the list back up; drop "parked" styles (a future `next_shopify_price_review`). Shopify only, positive sales only. → price **up** / harvest.
1b. **LOSERS:** the mirror — slow / stuck stock to price **down** ("cut to get it moving", might become a winner). Candidates = in stock, un-parked, SHP. Measure over **90 days** (a longer lens than winners' 30d, so a slow-but-alive style isn't mistaken for dead). `cover = weeks-to-clear = stock × (days/7) / u_win`. Membership = **DEAD** (`u_win = 0`) OR **SLOW** (`cover ≥ coverWeeks`, default 26 ≈ 6 months — the gate that keeps healthy sellers off the list). Order: **DEAD cluster first** (flagged "no recent sales"), then SLOW; within each, **most stock first** (stock at risk). Seasonality is the human's segment choice (not filtered here); size-residue is left in and parked with a long review. Route: `pricing-losers.js`.
2. **Drill-down (decide):** header (price/rrp/cost/stock/margin) + a **pricing timeline** (one row per distinct price sold at, newest first — latest era on top, with units AND **pace = /wk**) + a collapsible **size curve**.
   - **Pace matters** because total units mislead across periods of different length. `per_wk = units / weeks`, `weeks = max(span_days, 7)/7`, `span_days = last_sale − first_sale` for that price. Floor at 1 week so a tiny era doesn't show a wild number. Computed **app-side**.
   - **Honest caveat:** for seasonal Birkenstock, units rising as price rose can be the season arriving, not the price working. The clean signal for going higher is a price step where **pace held**.
   - **Size curve** is a guardrail before a **cut** (don't misread a sold-out core, e.g. 38/39 gone, as dead demand). Show ALL sizes with 0 for sold-out.
3. **Set price + review:** user sets a new price AND a review period together. Suggested defaults by move: raise ~7d, cut ~14d, hold ~30d (suggest, let them change).

**Review cooldown (`next_shopify_price_review`, a date):** while it's a future date the style is hidden from triage. A review period is **optional** on a price change (owner decision — a "None" chip, mirroring the Add/Modify price editor): pick a period to park the style, or None to leave the review date untouched. `changed_by` = the logged-in user's display name. `reason_code` is left NULL; `reason_notes` carries an **optional** free-text note from the operator (blank if none).

## Writes (must be exact — wrap each in `withTransaction`)

- **W1 — apply price:** `reviewDays` is **optional** (if supplied must be an integer `>= 1`; omitted/None = leave the review date untouched). Bounds (server-side, never trust client): **block** `< cost`; **allow-but-flag** `> rrp`. (The `minshopifyprice`/`maxshopifyprice` bounds were removed per owner — unused.) Then, atomically: `UPDATE skusummary SET shopifyprice=$price_string[, next_shopify_price_review=CURRENT_DATE+$reviewDays]` (the review clause only when a period was given) **and** `INSERT price_change_log (groupid,'SHP',old,new,NULL,$note,changed_by)` (`$note` = optional operator note, else ''). **Do NOT set `shopifychange`** — after the transaction commits, W1 pushes the new price to Shopify **immediately** via the Admin API (`utils/shopify.js → pushIfLive`, same as the Add/Modify module). Setting the nightly-sync flag too would double-push. The push is best-effort (never rolls back the DB write); on failure the response carries `shopify:{pushed:false,…}` and the operator re-Applies (`productSet` is idempotent). No nightly fallback (owner decision: "pure direct push"). **Same for Google:** right after the Shopify push, W1 also pushes the new price to Google Merchant Center's Content API (`utils/googleMerchant.js → pushIfLive`, shelling out to `scripts/google-price-push/push_google_price.py` — same "reuse the Python auth path" pattern as the Amazon-upload helper). Otherwise Google Shopping/ads would show the old price until the next nightly `C:\scripts\merchant-feed\merchant_feed.py --upload` cron run (that script only regenerates + re-uploads the feed once a day, at 3:30am BST). Only fires when the style is live on Google (`skusummary.googlestatus=1 AND shopify=1`); best-effort, never rolls back the DB write; failure surfaces as `google:{pushed:false,…}` and the operator re-Applies. A groupid can map to several Google product ids (one `skumap.googleid` per size) — the helper pushes them all in one Content API `custombatch` call (a single HTTP round-trip, not one request per size — the earlier one-call-per-size version was noticeably slower).
- **W2 — park only:** `UPDATE skusummary SET next_shopify_price_review=CURRENT_DATE+$reviewDays`. No price change, no log row, `shopifychange` untouched.

## Schema landmines (legacy DB — respect these)

- **Price columns on `skusummary` (`shopifyprice`, `cost`, `rrp`, `minshopifyprice`, `maxshopifyprice`) are `character varying`, not numbers**, and can contain junk (real case: `maxshopifyprice='RRP'` in 37 rows). Read them via `utils/sql.js → safeNumeric(col)` (returns NULL on non-numeric), never a bare `::numeric` (it throws). **Write `shopifyprice` as a 2dp STRING** (e.g. `'36.95'`), rounded to 2dp.
- **Never read stock from `skusummary.stockvariants` / `variants`** (stale, can be a year old). Current sellable stock = `localstock` where `ordernum='#FREE' AND COALESCE(deleted,0)=0 AND qty>0`.
- **`localstock` holds in-stock rows only** — a sold-out size has no row. For the full size range use **`skumap`** (one row per variant; size = `RIGHT(code,2)`), LEFT JOIN the sellable stock, default 0.
- **Size = `RIGHT(code,2)`** (EU size, by design).
- **`skusummary.colour` is an overloaded segmentation tag** (e.g. "Mocha" filed under "Brown") — ambiguous. Use `title.shopifytitle` for a human-readable name.
- **Dates:** legacy `created`/`updated` on `skusummary` (and `updated` on other tables) are TEXT (`'YYYYMMDD HH24:MI:SS'`, Europe/London). A proper **`skusummary.created_at timestamptz DEFAULT now()`** was added for going-forward use — new inserts get it; the 272 pre-existing rows are NULL (not backfilled). Prefer `created_at` for real date logic; keep writing the legacy text stamps too for compatibility. Same pattern on `price_change_log`: the legacy `change_date` (bare DATE) was joined by **`changed_at timestamptz DEFAULT now()`** (added 2026-07-10) for exact time-of-day — new inserts populate it via the default; older rows are NULL. `pricing-history` surfaces it as `changed_time` (Europe/London HH:MM). Migration: `bcweb-server/migrations/`.

## Conventions (full detail in docs/API-RULES.md)

- **Every API response is HTTP 200 + a `return_code`** (`SUCCESS` or an error code like `MISSING_FIELDS`, `NOT_FOUND`, `UNAUTHORIZED`, `SERVER_ERROR`). **Never** 4xx/5xx for API-level errors. Success adds data fields; errors add `message`.
- **One route file per endpoint** in `routes/` (kebab verb-noun), each starting with a structured header block and heavily commented (explain *why*). `utils/apiLogger.js` logs every route.
- **JWT carries only `{ id }`.** `middleware/verifyToken.js` looks the user up per request → `req.user = { id, display_name }`. `changed_by` is always resolved server-side, never sent by the client. JWT config in `config/config.js`.
- **DB:** central pool in `database.js` (`const { query } = require('../database')`); `utils/transaction.js` `withTransaction` for W1/W2; parameterised queries only (no string interpolation of user input); no N+1.
- **Frontend API client (`src/lib/api.ts`) never throws on API-level errors** — returns `{ success, data?, error?, return_code? }`; only genuine network failures throw. Auth state in `src/contexts/AuthContext.tsx`.
- **No hard-coded secrets** — everything via `.env`; flag if a new var is needed.

## Endpoints

`POST /login` · `GET /pricing-segments` · `GET /pricing-triage?segment&days?&limit?` (WINNERS) · `GET /pricing-losers?segment&days?&limit?&coverWeeks?` (LOSERS) · `GET /pricing-drill?groupid&days?` · `GET /pricing-history?groupid&limit?` (drill report: recent price-change log, most-recent-N, lazy) · `GET /pricing-sales?groupid&limit?` (drill report: recent raw sales w/ sold price, most-recent-N, lazy) · `GET /pricing-find?term` · `POST /pricing-apply {groupid,newPrice,reviewDays}` (W1) · `POST /pricing-park {groupid,reviewDays}` (W2) · `POST /pricing-park-bulk {groupids[],reviewDays}` (bulk W2 — batch "just set review" over a selection, set-based UPDATE; the Shopify mirror of `/amz-review`) · `GET /health`. All except login/health require `verifyToken`. **Bulk pricing on the WINNERS/LOSERS lists:** the segment lists carry a collapsible `BulkActionBar` (`bcweb-web/src/components/BulkActionBar.tsx`) that mirrors the drill's price-setter across a *selection* — a **relative** price move built from the same denomination nudges (applied to each row's own current price; hence `pricing-triage`/`pricing-losers` now return `price`) and/or a review chip. A bulk **price** change deliberately **loops `POST /pricing-apply` (W1) client-side, one style at a time** (so each style's live Shopify + Google push still fires and per-item bounds still apply) rather than a batch write; a bulk **review-only** uses `/pricing-park-bulk`. Below-cost items are just skipped+reported (no special UI).

**Amazon Pricing** (a SKU-grain mirror of the Shopify flow — segment picker → WINNERS|LOSERS|ALL lists → per-SKU drill; see `docs/amz-pricing-spec.md`): `GET /amz-segments` · `GET /amz-winners?segment&days?&limit?` · `GET /amz-losers?segment&days?&limit?&coverWeeks?` · `GET /amz-all?segment` · `GET /amz-drill?code` · `GET /amz-history?code&limit?` · `GET /amz-sales?code&limit?` · `GET /amz-find?term` · `POST /amz-apply {code,newPrice,note?,reviewDays?}` (W-A1) · `GET /amz-basket` (rebuild today's upload basket from the log). Key differences from Shopify: **SKU-grain** (each size = its own Amazon price/row; `code` not `groupid`); reads `amzfeed` (FBA-only, **READ ONLY** — refreshed nightly from Amazon, never written) for live price/stock, `sales` where `channel='AMZ'` for velocity; **no live push** — W-A1 only logs to `amz_price_log` (blocks `< cost+fbafee` floor, flags `> rrp`), and the price reaches Amazon via a **client-built one-file Seller Central upload**. **Optional review/park** on apply (`reviewDays?`, mirrors Shopify W1 exactly: omitted/null = None = leave `next_amz_price_review` untouched; an integer parks the SKU out of the winners/losers queue). The upload basket (`src/contexts/AmzBasketContext.tsx`) is a **DURABLE, TEAM-WIDE view of recent price changes** — a rolling **12h** window across **all operators** (so whoever is at the desk can upload a colleague's pending change) — rebuilt on load from `amz_price_log` via `GET /amz-basket` (latest price per SKU, any operator, `changed_at >= now() - 12h`) — so it survives a browser close and the file can be regenerated any time (the Seller Central upload is idempotent, so no upload-state tracking and no "clear"; the rolling window self-clears). Needs `amz_price_log.changed_at timestamptz` (migration `20260713_amz_price_log_changed_at.sql`, applied; older rows NULL, mirrors `price_change_log.changed_at`). Still never writes `amzfeed`. Winners = in-stock, sold in 30d; Losers = DEAD (no AMZ sale in 14d) or SLOW (cover ≥ 16wk). Amazon "margin" is **net** (price − cost − FBA fee). **Bulk pricing** works the same as Shopify (shared `BulkActionBar`): a bulk price move **loops `POST /amz-apply` (W-A1) per ticked SKU** (queuing each into the upload basket), and bulk review-only reuses the existing batch `POST /amz-review` — so there is no new Amazon bulk endpoint.

**Order Status** (supplier orders in `orderstatus`; see `docs/order-status-lifecycle.docx` and `docs/orders/legacy/order-request.txt`): `GET /order-status-suppliers` (picker, both stages) · `GET /order-status-to-place?supplier` · `GET /order-status-find?supplier&term` · `POST /order-status-add {supplier,code,qty,ordertype?}` · `POST /order-status-place {ordernums[]}` · `POST /order-status-unplace {ordernums[]}` · `GET /order-status-list?supplier` (ON ORDER batches) · `POST /order-status-switch-type` · `POST /order-status-archive` · `POST /order-status-adjust-qty`. Scope is **supplier** orders only — `ordertype` 2 (local) and 3 (Amazon); customer orders (`ordertype=1`) are never touched.

The module is **two stages of one lifecycle — Chosen → Placed → Arrived — split on the `orderdate` marker** (`utils/orderStatus.js` owns the predicates so no route can drift):
- **TO PLACE** (`orderdate = ''`): the legacy PowerBuilder request screen INSERTS the rows and deliberately leaves `orderdate` blank — the goods are chosen but **not yet bought from the supplier**. Nothing is coming until someone acts. The build sheet aggregates to **one row per SKU across every `createddate` AND both ordertypes** (the supplier gets one line for 2 pairs; Amazon-vs-local is our internal routing, not theirs), then the operator exports a **CSV of `barcode,qty,code`** (client-built, `PlaceOrderSheet.tsx`) and marks it ordered.
- **ON ORDER** (`orderdate <> ''`): genuinely with the supplier — the chase list. Batches key on the **`orderdate` stamp**, not `createddate`, and age counts from it (days the SUPPLIER has had it).

**Adding a line (`/order-status-find` + `/order-status-add`)** covers "the supplier offered a deal while I'm placing the order" without going back to PowerBuilder. **TO PLACE only** — and adding to a *placed* batch isn't blocked so much as inexpressible: every row this route writes is un-placed by construction. That's deliberate; a row inserted into a placed batch would claim to be part of a placement that already happened, carrying an `orderdate`/PO the supplier's real order doesn't contain, so Goods In would reconcile against a receipt nobody ordered. The code is **picked, never typed** (typeahead over `skumap`, scoped to `skumap.supplier = $supplier`), which makes a typo, a non-existent code and another supplier's style impossible by construction. Unlike `+` in `/order-status-adjust-qty` — which is safe because it CLONES a row and inherits ~40 legacy columns free — a new code has no template, so `order-status-add.js` sets every field explicitly per `docs/orders/legacy/order-request.txt` (`qty=1`, `batch='0'`, `channel='MANUAL'`, `orderdate=''`, `'x'` in every shipping/customer text column, `ukd`/`othersupplier` by the UKD rule). `fnsku` joins **`amzfeed.code`, NOT `amzfeed.sku`** (`sku` is the Amazon seller-SKU, a different string — joining on it silently yields no fnsku).

**W-O1 — place** (`POST /order-status-place`, in `withTransaction`): sets `orderdate` **and** `updated` to `to_char(now() AT TIME ZONE 'Europe/London','YYYYMMDD HH24:MI:SS')` — the legacy TEXT format, so a web-placed row is indistinguishable from a legacy one — plus a minted `ponumber` of `BC-YYYYMMDD-NNN` shared by every unit in the placement. `nextPoNumber()` takes a **transaction advisory lock** and derives the counter from `orderstatus` + `orderstatus_archive`, so it can't collide or restart; the `BC-` prefix keeps it disjoint from legacy supplier-allocated 6-digit numbers. The UPDATE carries `AND COALESCE(orderdate,'')=''`, making a double-click or retry a no-op rather than a restamp that would reset the delivery age and lose the original PO. **Never touches `arrived`/`arriveddate`** (goods-in stays legacy) and inserts no rows. **W-O2 — unplace**: the undo, guarded by `ponumber LIKE 'BC-%'` so only orders THIS app placed can be un-ordered; resets `orderdate` to the **empty string, not NULL**.

**Landmines specific to this module:** `orderstatus.orderdate` is `character varying` and an un-placed row holds an **EMPTY STRING** — but it can also be **NULL** in the wild (PowerBuilder converts `''` → NULL on save), so always test it as `COALESCE(orderdate,'') = ''`; neither `IS NULL` nor `= ''` alone is correct. `orderstatus` has **one row per physical unit** (`qty` always 1, duplicated lines), so quantities are `COUNT(*)` and +/- means insert/archive whole rows. Order cost comes from **`skusummary.cost` via `safeNumeric`, never `skumap.cost`** (blank on ~1200 of 2046 live SKUs, `'1000'` placeholders on ~370 more). Barcode is `skumap.ean` **with the legacy trailing `B` stripped** (an Excel guard, not part of the EAN) — and the ~61 SKUs with no EAN are blocked from the CSV rather than exported blank. Dates are cast to text **in SQL**; never hand a pg `DATE` to `toISOString()` (node-postgres parses it as local midnight, so BST shifts the day back one — this bit `order-status-list.js` before the split).

## Working notes / cautions

- The API points at the **LIVE production DB**. Reads are safe; **W1/W2 write real product rows.** When testing writes, wrap in a manual `BEGIN … ROLLBACK` rather than committing to a real style, unless the owner OKs a live apply-and-restore.
- Git is not initialised yet (owner's choice). Single `.gitignore` at repo root.
- Don't invent pricing logic — the process above is the source of truth. Out of scope for v1: Shopify API calls, Amazon/other channels, bulk edits, other modules.
